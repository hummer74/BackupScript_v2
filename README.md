# Аудит репозитория BackupScript_v2

**Репозиторий:** https://github.com/hummer74/BackupScript_v2
**Дата аудита:** 21.04.2026
**Назначение:** резервное копирование виртуальных машин Hyper-V

---

## 1. Обзор репозитория

Репозиторий содержит набор PowerShell-скриптов для автоматического резервного копирования виртуальной машины Hyper-V (VM-имя: `1c-azimut`). Система состоит из трёх основных скриптов, вспомогательных файлов конфигурации почты и XML-файлов для интеграции с планировщиком задач Windows (Task Scheduler).

### Состав файлов

| Файл | Назначение |
|------|-----------|
| `Backup-HyperV.ps1` | Главный скрипт: проверка предусловий, остановка ВМ, экспорт, сжатие в 7z, ротация |
| `Rotate-Weekly.ps1` | Еженедельная ротация: копирует последний дневной архив в недельное хранилище |
| `Rotate-Monthly.ps1` | Ежемесячная ротация: копирует последний недельный архив в месячное хранилище |
| `MailConfig.ps1` | Конфигурация SMTP для отправки уведомлений об ошибках |
| `MailFunctions.ps1` | Функция `Send-ErrorNotification` для отправки писем |
| `00 Hyper-V Backup.xml` | XML-профильт планировщика задач: ежедневный бэкап в 20:30 |
| `02 Hyper-V Rotate Weekly.xml` | XML-профильт: еженедельная ротация по воскресеньям в 00:00 |
| `03 Hyper-V Rotate Monthly.xml` | XML-профильт: ежемесячная ротация 1-го числа в 01:00 |

### Структура хранения бэкапов

```
F:\HyperV-B\
├── 00_Daily\       — ежедневные архивы (хранятся 5 штук)
├── 01_Weekly\      — еженедельные копии (хранятся 4 штуки)
└── 02_Monthly\     — ежемесячные копии (хранятся 6 штук)
```

Временная папка для экспорта: `E:\Z-TEMP\HyperV-Backup`. Лог-файл: `E:\HyperV-Backup.log`.

## 2. Пошаговое описание работы главного скрипта (`Backup-HyperV.ps1`)

### Фаза 1: Инициализация и проверка предусловий

Скрипт начинается с загрузки конфигурации из блока `# === CONFIGURATION ===` в первых строках. Затем подключаются внешние файлы `MailConfig.ps1` и `MailFunctions.ps1` через оператор «точка» (`.`), что загружает переменные и функции напрямую в текущую область видимости.

**Шаг 1. Проверка наличия 7-Zip.** Скрипт проверяет существование файла `C:\Program Files\7-Zip\7z.exe`. Если 7-Zip не установлен — записывается ошибка и скрипт завершается.

**Шаг 2. Проверка существования виртуальной машины.** Через командлет `Get-VM` проверяется наличие ВМ с именем `1c-azimut` на текущем хосте Hyper-V. Если ВМ не найдена — ошибка и выход.

**Шаг 3. Создание каталогов.** Скрипт проверяет наличие трёх каталогов (`F:\HyperV-B`, `F:\HyperV-B\00_Daily`, `E:\Z-TEMP\HyperV-Backup`) и при необходимости создаёт их через `New-Item`. Ошибка создания любого каталога приводит к завершению скрипта.

**Шаг 4. Проверка свободного места на дисках.** Скрипт вычисляет суммарный размер VHD-дисков виртуальной машины, умножает его на 1.1 (10 % запас) и проверяет:
- Диск `E:` (временный экспорт) — достаточно ли места для raw-экспорта ВМ
- Диск `F:` (финальное хранилище) — достаточно ли места для сжатого архива

Если хотя бы одна проверка не пройдена — скрипт завершается с отправкой уведомления на почту.

### Фаза 2: Выполнение резервного копирования

Если все предусловия выполнены, начинается основная фаза.

**Шаг 5. Остановка виртуальной машины.** Если ВМ была в состоянии «Running», скрипт запускает корректное завершение работы через `Stop-VM` с таймаутом 600 секунд (10 минут). Если за это время ВМ не остановилась — выполняется принудительное выключение (`Stop-VM -Force`) с дополнительным таймаутом 60 секунд. Если и он не помогает — скрипт выбрасывает исключение и прерывается, чтобы не повредить данные.

**Шаг 6. Экспорт ВМ.** Через командлет `Export-VM` виртуальная машина экспортируется целиком (конфигурация, контрольные точки, диски) во временную папку `E:\Z-TEMP\HyperV-Backup\1c-azimut-<дата_время>`.

**Шаг 7. Немедленный запуск ВМ.** Сразу после завершения экспорта — **до начала сжатия** — виртуальная машина запускается обратно через `Start-VM`. Это критически важное решение: простой ВМ минимизируется до времени экспорта, а не до времени экспорта + сжатия. Скрипт ждёт перехода в состояние «Running» до 60 секунд. Если ВМ не стартовала — ошибка записывается, но выполнение продолжается (сжатие всё равно нужно выполнить).

**Шаг 8. Сжатие в 7z.** Экспортированная папка архивируется через 7-Zip с уровнем сжатия `mx5` (средний). Команда: `7z a -t7z -mx5 <архив> <файлы> -bsp1` (флаг `-bsp1` включает вывод прогресса на консоль). Результат — файл `F:\HyperV-B\00_Daily\1c-azimut-<дата_время>.7z`.

**Шаг 9. Ежедневная ротация.** Скрипт получает список всех `.7z`-файлов в `00_Daily`, сортирует их по дате изменения (от новых к старым) и удаляет все файлы, начиная с шестого (то есть оставляет только 5 самых новых). Ротация выполняется **только** при успешном создании архива.

### Фаза 3: Завершение и очистка (блок `finally`)

**Шаг 10. Удаление временных данных.** Временная папка экспорта удаляется через `Remove-Item -Recurse -Force`.

**Шаг 11. Финальная проверка состояния ВМ.** Если ВМ до начала бэкапа была запущена, скрипт проверяет, работает ли она сейчас. Если нет — пытается запустить принудительно. Это страховка на случай, если сжатие завершилось ошибкой, и ВМ не была запущена на Шаге 7.

**Шаг 12. Отправка уведомления об ошибках.** Если за время работы скрипта были зафиксированы хотя бы одна ошибка — отправляется письмо через `Send-ErrorNotification`.

## 3. Анализ интеграции с планировщиком задач (XML)

### Общий профиль всех трёх задач

Все три XML-файла описывают задачи, которые:
- Созданы пользователем `HARD_SERVER\opossum`
- Работают от имени этого пользователя с сохранённым паролем (уровень `HighestAvailable` — наивысшие привилегии)
- Запускают PowerShell с параметрами `-NoProfile -ExecutionPolicy Bypass`
- Выполняются только когда пользователь вошёл в систему (флаг `LogonType` = `S-1-5-...-1001` означает интерактивный вход)
- При пропуске запуска — не компенсируют его (`StartWhenAvailable = false`), а просто игнорируют (`ExecutionTimeLimit = PT3H` или `PT1H`, `MultipleInstancesPolicy = IgnoreNew`)

### Расписание

| Задача | Триггер | Время запуска |
|--------|---------|--------------|
| `00 Hyper-V Backup` | Ежедневно | 20:30 |
| `02 Hyper-V Rotate Weekly` | Каждое воскресенье | 00:00:01 |
| `03 Hyper-V Rotate Monthly` | 1-е число каждого месяца | 01:00:01 |

### Выявленные проблемы XML-конфигурации

**Проблема 1: Триггеры пересекаются по времени.** Недельная ротация запускается в воскресенье в 00:00:01, а ежемесячная — 1-го числа в 01:00:01. Если 1-е число выпадает на воскресенье (что бывает примерно раз в 7 месяцев), обе задачи выполнятся с интервалом в один час. Это само по себе не критично, но недельная ротация может скопировать ещё не полностью обновлённый дневной архив, если ежедневный бэкап по какой-то причине задержался.

**Проблема 2: Задачи привязаны к конкретному пользователю и его паролю.** Все задачи настроены на выполнение от имени `HARD_SERVER\opossum`. Если смена пароля учётной записи произойдёт — задачи перестанут запускаться автоматически. Для серверных задач принято использовать учётную запись `SYSTEM` или отдельную сервисную учётную запись с фиксированным паролем.

**Проблема 3: Задачи требуют интерактивного входа пользователя.** Идентификатор входа `S-1-5-21-175866239-2976161984-1870451039-1001` — это SID пользователя `opossum`. Задачи с таким типом входа выполняются **только** когда пользователь вошёл в систему. Если сервер перезагрузится и никто не выполнит вход — бэкап не запустится. Для серверного резервного копирования это критическая уязвимость.

**Проблема 4: Путь к скриптам не совпадает с репозиторием.** В XML-файлах указан путь `C:\Script\Backup-HyperV.ps1`, а в самом скрипте `Backup-HyperV.ps1` в строке подключения `MailConfig.ps1` указан путь `C:\Script\MailConfig.ps1`. При этом репозиторий предполагает, что скрипты лежат в одной папке. Это расхождение может вызвать ошибки при развёртывании на другой сервер.

**Проблема 5: `MultipleInstancesPolicy = IgnoreNew`.** Если предыдущий запуск бэкапа задерживается (например, ВМ долго выключается), новый запуск не начнётся, а предыдущий запуск в 20:30 может занять более 3 часов (лимит выполнения `PT3H`). Запуск в 20:30, в принципе, разумное время, но при больших размерах VHD (сотни гигабайт) экспорт + сжатие могут длиться дольше.

## 4. Скрипты ротации

### Еженедельная ротация (`Rotate-Weekly.ps1`)

Скрипт берёт самый свежий архив из `00_Daily` и копирует его в `01_Weekly`. Если файл с таким именем уже существует — пропускает. Затем оставляет только 4 самых новых недельных архива. По сути, это «горизонт» хранения в 4 недели.

Особенность: скрипт не проверяет возраст дневного архива. Он просто берёт последний по дате изменения. Если ежедневный бэкап не выполнялся несколько дней, недельная ротация скопирует самый последний (возможно, очень старый) архив.

### Ежемесячная ротация (`Rotate-Monthly.ps1`)

Аналогично недельной: берёт последний архив из `01_Weekly`, копирует в `02_Monthly`, оставляет 6 самых новых. Это даёт «горизонт» хранения в 6 месяцев.

Каскадная схема: `Дневной → Недельный → Месячный`. Идея понятная и распространённая, но реализация имеет слабое место — отсутствие проверки свежести копируемого архива.

## 5. Анализ ротации: сценарий «3 запуска за один день»

### Исходные данные

- `$DaysToKeep = 5` — оставлять 5 самых новых архивов в `00_Daily`
- Ротация считается **по количеству файлов**, а не по количеству дней
- Скрипт запускается каждый день в 20:30

### Что произойдёт, если сегодня скрипт запустили 3 раза подряд

Предположим, текущая ситуация в `00_Daily` (5 архивов, по одному на каждый из последних 5 дней):

```
День 1 (старый) → День 2 → День 3 → День 4 → День 5 (новый)
```

После тройного запуска сегодня:

```
День 2 → День 3 → День 4 → День 5 → Сегодня_12:00 → Сегодня_14:00 → Сегодня_20:30
```

Но ротация отсекает всё, начиная с 6-го файла:

```
День 3 → День 4 → День 5 → Сегодня_12:00 → Сегодня_14:00
```

Архив **Дня 2 удалён**. Вместо 5 дней истории у нас осталось только 2 уникальных дня (Дни 4 и 5), а остальные 3 слота заняты архивами с одного и того же дня.

### Через две недели при ежедневных запусках

Если после тройного запуска скрипт продолжает работать нормально (1 раз в день), ротация будет выглядеть так:

| День после инцидента | Архивы в 00_Daily (старый ← новый) |
|---|---|
| День 0 (сегодня, 3 запуска) | Д3, Д4, Д5, С_12:00, С_14:00 |
| День 1 | Д4, Д5, С_12:00, С_14:00, С_20:30 |
| День 2 | Д5, С_12:00, С_14:00, С_20:30, Д+1 |
| День 3 | С_12:00, С_14:00, С_20:30, Д+1, Д+2 |
| День 4 | С_14:00, С_20:30, Д+1, Д+2, Д+3 |
| День 5 | С_20:30, Д+1, Д+2, Д+3, Д+4 |

Полное «восстановление» горизонтов наступит только на **6-й день** после инцидента, когда последний «лишний» архив от тройного запуска (`С_20:30`) вытеснится из пятёрки.

**Вывод:** Тройной запуск сокращает эффективный горизонт хранения с 5 дней до 2 на первые несколько дней. «Волна» лишних архивов проходит через ротацию за ~6 дней. Это не катастрофично, но при более частых повторных запусках (например, отладка скрипта) можно полностью потерять историю бэкапов за несколько предыдущих дней.

### КРИТИЧЕСКИЕ

| # | Файл | Описание |
|---|------|----------|
| 1 | `MailConfig.ps1`, строка 5 | **Пароль хранится в открытом виде.** Строка `$SmtpPassword = 'pass'` содержит реальный пароль от ящика `dan@dex.com` и опубликована в публичном GitHub-репозитории. Любой человек, скачавший репозиторий, видит пароль. Пароль необходимо немедленно сменить, а в коде использовать защищённый хранение (Credential Manager, зашифрованный файл, переменные окружения). |

| # | Файл | Описание |

При множественных запусках в один день горизонт хранения сокращается. Рекомендуется переименовать в `$CopiesToKeep` или добавить логику «один архив в день». |

| # | Файл | Описание |
|---|------|----------|
| 5 | `MailFunctions.ps1` | **Используется устаревший класс `Net.Mail.SmtpClient`.** В .NET этот класс помечен как устаревший (deprecated) начиная с .NET Framework 4.5. Рекомендуется использовать `System.Net.Mail.MailMessage` с `SmtpClient` из более нового API или модуль `Send-MailMessage` PowerShell. |
| 6 | `Rotate-Weekly.ps1`, `Rotate-Monthly.ps1` | **Нет проверки возраста архивов.** Скрипты ротации копируют последний архив без проверки, насколько он свежий. Если ежедневный бэкап не выполнялся несколько дней, ротация скопирует устаревший файл. |

### НИЗКИЕ

| # | Файл | Описание |
|---|------|----------|
| 9 | `MailFunctions.ps1`, строка 24 | **Эмодзи в выходном потоке.** Строки `✅ Email sent...` и `❌ Failed to send...` содержат эмодзи, которые могут некорректно отображаться в логах Windows или при перенаправлении вывода. |
| 10 | `Backup-HyperV.ps1` | **Нет валидации `$VMName`.** Если имя ВМ указано с опечаткой (например, `1c-azimut ` с пробелом), скрипт не найдёт машину, но не даст понятного подсказки об этом — просто запишет общую ошибку. |
| 11 | `MailConfig.ps1`, строка 7 | **Закомментированные адреса получателей.** Строка содержит `# mvu@ural.ru, mpn@ural.ru` — видимо, актуальные получатели закомментированы. Текущий список получателей (`john@mail.com, dan@dex.com`) может быть неактуальным. |

## 7. Рекомендации

### Срочно (сделать немедленно)

1. **Сменить пароль** от ящика `dan@dex.com`, так как он опубликован в открытом репозитории.
2. **Исправить `$MailFrom`** на валидный email-адрес.
3. **Перевести задачи планировщика** на выполнение от имени `SYSTEM` вместо пользователя `opossum`. Для этого в XML нужно заменить `<UserId>` на `<UserId>S-1-5-18</UserId>` (SID SYSTEM) и удалить `<LogonType>`.
4. **Сделать репозиторий приватным** или убрать `MailConfig.ps1` из-под контроля версий (добавить в `.gitignore` и создать шаблон `MailConfig.ps1.example`).

### Краткосрочные улучшения

5. **Переименовать `$DaysToKeep`** в `$CopiesToKeep` для точности, либо переписать ротацию на основе дат (оставлять один архив за каждый из последних N дней).
6. **Добавить проверку свежести** в скрипты ротации: если последний дневной архив старше 48 часов — отправлять предупреждение.
7. **Вынести общие функции** (`Log`, `Add-Error`, `Send-ErrorNotification`) в модуль `Common.ps1` и подключать через «точку» во всех скриптах.
8. **Увеличить лимит выполнения** ежедневного бэкапа или убрать его (`ExecutionTimeLimit = PT0S` означает «без лимита»).

### Долгосрочные улучшения

9. Использовать защищённое хранение учётных данных (Windows Credential Manager, DPAPI, или хотя бы `ConvertFrom-SecureString`).
10. Добавить проверку целостности архивов (тест `7z t` после создания).
11. Реализовать мониторинг: если бэкап не выполнялся более 24 часов — отправлять предупреждение.
12. Удалить эмодзи из выходного потока скриптов.

## 8. Резюме

Система резервного копирования в целом работоспособна и решает свою задачу. Логика «остановка → экспорт → немедленный запуск → сжатие» разумна и минимизирует простой виртуальной машины. Каскадная схема ротации (день → неделя → месяц) обеспечивает хороший баланс между надёжностью и занимаемым местом.

Однако есть несколько серьёзных проблем. Критическая — хранение пароля в открытом виде в публичном репозитории. Это необходимо исправить немедленно. Высокоприоритетные — невалидный адрес отправителя (письма просто не отправятся) и привязка задач к интерактивному входу пользователя (бэкап не запустится после перезагрузки без входа). Наконец, ротация по количеству файлов, а не по дням, создаёт неочевидное поведение при множественных запусках, что может привести к преждевременной потере старых архивов.
